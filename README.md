# Indic Tokenizer + Decoding-from-Scratch

A from-scratch NLP pipeline covering the fundamentals that library calls usually hide:
**subword tokenization, a small Transformer language model, and four text-decoding
strategies** — all implemented and evaluated on a Hindi (Devanagari) corpus.

Built to directly demonstrate the skills listed in Apple's NLP/ML Researcher job
descriptions: tokenization, language modeling, decoding, and multilingual (Indic)
text handling — with actual numbers, not just "it runs."

## Why this project exists

Most student NLP portfolios call `tokenizer.encode()` and `model.generate()` without
knowing what happens inside. This project deliberately implements both pieces from
scratch, then compares them against the standard library implementation, so every
design decision is defensible in an interview.

## Project structure

```
indic-tokenizer-decoding/
├── data/
│   └── generate_corpus.py     # builds an original, synthetic Hindi sentence corpus
│   └── corpus_hi.txt          # generated corpus (1200 sentences)
├── tokenizer/
│   ├── train_sentencepiece.py # production BPE tokenizer (SentencePiece library)
│   └── bpe_from_scratch.py    # BPE algorithm implemented from scratch, no libraries
├── model/
│   ├── lm.py                  # TinyGPT: decoder-only Transformer, built from scratch
│   └── train.py                # training loop
├── decoding/
│   └── strategies.py          # greedy, beam search, top-k, top-p — all from scratch
├── generate.py                 # CLI: generate text with every strategy, side by side
├── eval/
│   └── evaluate.py             # perplexity, tokenizer efficiency, decoding diversity
└── requirements.txt
```

## Why a synthetic corpus?

The corpus (`data/corpus_hi.txt`) is generated by `generate_corpus.py` using
templated composition over hand-picked Hindi vocabulary (subjects, verbs, places,
topics) — **not scraped or copied from any existing text.** This sidesteps
copyright entirely and gives full control over vocabulary size for a fast, CPU-only
demo. Swapping in a real corpus (e.g. AI4Bharat's IndicCorp or Samanantar) requires
no pipeline changes — only `data/corpus_hi.txt` changes.

## 1. Tokenization

Two implementations, on purpose:

| | `train_sentencepiece.py` | `bpe_from_scratch.py` |
|---|---|---|
| Approach | Google's SentencePiece library (BPE mode) | Manual pair-frequency merging, ~150 lines |
| Vocab size | 400 | 202 (150 merges) |
| Purpose | Production-grade, what you'd actually deploy | Proves understanding of the algorithm itself |

Both correctly handle Devanagari conjuncts and out-of-vocabulary words by falling
back to smaller subword pieces — e.g. the unseen word "खेलते" gets split into
`खेल` + `त` + `े` rather than becoming `<unk>`.

**Result:** 0.00% unknown-token rate on held-out sentences, 1.17 subword pieces
per word on average (see `eval/evaluate.py` output).

## 2. Language model

`model/lm.py` implements a decoder-only Transformer (`TinyGPT`) from scratch:
causal self-attention, positional embeddings, and feed-forward blocks — the same
architecture pattern used in GPT-family models, just intentionally small
(2 layers, 128-dim, ~500K params) so it trains in seconds on CPU.

**Training result** (15 epochs, `model/train.py`):

| Epoch | Val Loss | Val Perplexity |
|---|---|---|
| 1 | 5.41 | 224.5 |
| 5 | 3.49 | 32.8 |
| 10 | 2.57 | 13.0 |
| 15 | 2.01 | **7.47** |

## 3. Decoding strategies (`decoding/strategies.py`)

All four implemented manually on raw logits — no `model.generate()` anywhere:

- **Greedy** — always picks the single highest-probability token.
- **Beam search** (width=4) — tracks multiple candidate sequences by cumulative
  log-probability, keeping the globally best `k` at each step.
- **Top-k sampling** — samples from only the k highest-probability tokens.
- **Top-p / nucleus sampling** — samples from the smallest set of tokens whose
  cumulative probability exceeds p, adapting the candidate pool size per step.

### Measured diversity difference (self-BLEU, lower = more diverse)

| Strategy | Self-BLEU |
|---|---|
| Greedy | 0.543 |
| Beam (w=4) | 0.573 |
| Top-k (k=10) | 0.396 |
| Top-p (p=0.9) | 0.401 |

This matches the theory exactly: greedy/beam search are deterministic and produce
repetitive output across prompts, while top-k/top-p introduce controlled randomness
that measurably increases output diversity — with actual numbers to back it up.

## How to run

```bash
pip install -r requirements.txt

python3 data/generate_corpus.py          # generate the corpus
python3 tokenizer/train_sentencepiece.py # train + demo the production tokenizer
python3 tokenizer/bpe_from_scratch.py    # train + demo the from-scratch tokenizer
python3 model/train.py                    # train the language model (~1 min on CPU)
python3 generate.py                       # compare all 4 decoding strategies
python3 eval/evaluate.py                  # perplexity, tokenizer efficiency, diversity
```
