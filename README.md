# Multilingual Indic NLP Pipeline — Tokenization, Generation, Decoding, Safety, Compression

An end-to-end generative NLP pipeline covering **tokenization, language modeling,
decoding, safety-controlled generation, custom training internals, and model
compression** — across three Indic languages (Hindi, Telugu, Malayalam) with a
single shared model. Every core algorithm (BPE, Transformer, decoding strategies,
optimizer) is implemented from scratch, not just called from a library, so every
design decision is defensible in an interview.


## Project structure

```
indic-tokenizer-decoding/
├── data/
│   ├── generate_corpus.py       # original synthetic corpora: Hindi, Telugu, Malayalam

│   └── corpus_multilingual.txt  # combined, language-tagged: "<hi> ...", "<te> ...", "<ml> ..."
├── tokenizer/
│   ├── train_sentencepiece.py   # trains 1 shared multilingual + 3 per-language BPE tokenizers
│   └── bpe_from_scratch.py      # BPE algorithm implemented from scratch, no libraries

│   ├── lm.py                    # TinyGPT: decoder-only Transformer, built from scratch
│   └── train.py                 # trains TinyGPT on the multilingual tagged corpus
├── decoding/
│   └── strategies.py            # greedy, beam search, top-k, top-p — all from scratch
├── classifier/
│   ├── generate_labeled_data.py # synthetic sentiment-labeled data (Hindi)
│   ├── classifier_model.py      # BiLSTM text classifier, from scratch
│   ├── train.py                 # trains the classifier
│   └── safety_filter_demo.py    # wires the classifier into generation as a safety gate
├── custom_training/
│   ├── custom_optim.py          # LabelSmoothingLoss + SimpleAdamW, implemented from scratch
│   └── benchmark.py             # benchmarks both against PyTorch's built-ins, same task

│   └── quantize_and_benchmark.py # dynamic int8 quantization: size/latency/quality before-after

├── eval/evaluate.py             # perplexity, tokenizer efficiency, decoding diversity

```

## Why a synthetic corpus?

All corpora are generated by `generate_corpus.py` via templated composition over
hand-picked vocabulary per language — **not scraped from any existing text.*"
---

## 1. Tokenization — three languages, two implementations

| | Shared multilingual tokenizer | Per-language tokenizers | From-scratch BPE |
|---|---|---|---|
| File | `multilingual_bpe.model` | `hi_bpe` / `te_bpe` / `ml_bpe` | `bpe_from_scratch.py` |
| Vocab size | 800 | 400 each | 202 (150 merges) |
| Purpose | One model, all languages | Measure the multilingual tradeoff | Prove understanding of the algorithm |

**Key measured result — the multilingual tokenizer tradeoff:**

| Language | Dedicated tokenizer | Shared multilingual tokenizer |
|---|---|---|
| Hindi | 1.19 pieces/word | 1.53 pieces/word |
| Telugu | 1.29 pieces/word | 1.67 pieces/word |
| Malayalam | 1.30 pieces/word | 2.06 pieces/word |

This is the real, well-documented "vocabulary competition" effect in multilingual
NLP: sharing one vocabulary across scripts costs some per-language efficiency in
exchange for one unified model. Worth explaining explicitly — it's a real, known
tradeoff, not a bug in this pipeline.

## 2. Language model — one model, three scripts

`TinyGPT` (decoder-only Transformer, from scratch: causal self-attention + FFN
blocks, 3 layers, 160-dim, ~1.2M params) is trained on the **combined, language-tagged
corpus** — a single model handles Hindi, Telugu, and Malayalam via a language-tag
prefix (`<hi>`, `<te>`, `<ml>`), the same pattern real multilingual models like
mBART/mT5 use.

**Training result:** validation perplexity dropped from 105.5 (epoch 1) to **2.35**
(epoch 20) across all three languages combined.

**Generation result** — the same model, conditioned only on the tag, produces
grammatically coherent output in three different scripts:
```
<hi> राम बहुत खुश है और प्रकृति पसंद करता है।
<te> విద్యార్థి చాలా తెలివైన, మరియు విజ్ఞానశాస్త్రం ఇష్టపడతాడు.
<ml> കുട്ടികൾ ഇന്ന് സ്കൂൾ കളിക്കുന്നു.
```

## 3. Decoding strategies — from scratch

Greedy, beam search (width=4), top-k (k=10), and top-p/nucleus (p=0.9) — no
`model.generate()` anywhere. Measured with self-BLEU (lower = more diverse):

| Strategy | Self-BLEU |
|---|---|
| Greedy | 0.24 |
| Beam (w=4) | 0.25 |
| Top-k (k=10) | 0.20 |
| Top-p (p=0.9) | 0.24 |

## 4. Safety-controlled generation (text classifier + filter)

A BiLSTM sentiment classifier (`classifier/`), trained from scratch on synthetic
labeled Hindi sentences, is wired into generation as an accept/reject gate
(`safety_filter_demo.py`): generate a candidate → classify it → reject and
regenerate if it fails the policy (here: reject negative-sentiment output).

```
attempt 1: [rejected] (P(positive)=0.00)  ...
attempt 2: [rejected] (P(positive)=0.21)  ...
attempt 3: [ACCEPTED] (P(positive)=1.00)  राम बहुत खुश है और संगीत पसंद करता है।
```

This is a simplified but real instance of the generate → score → filter pattern
behind "safety-controlled text composition."

## 5. Custom loss function + optimizer, benchmarked against PyTorch's built-ins

`custom_training/custom_optim.py` implements:
- **LabelSmoothingLoss** — manual KL-divergence-based loss against a smoothed
  target distribution, instead of one-hot cross-entropy.
- **SimpleAdamW** — manual moving averages (m, v), bias correction, and decoupled
  weight decay — the full AdamW update rule, no `torch.optim`.

**Benchmark result** (`custom_training/benchmark.py`, same model, same data, same init):
- `SimpleAdamW` produces a val-loss trajectory **identical** to `torch.optim.AdamW`
  — strong evidence the reimplementation is mathematically correct.
- `LabelSmoothingLoss` shows a slightly *higher* held-out loss than plain
  cross-entropy (1.199 vs. 1.125 at epoch 10) — expected, since it deliberately
  keeps the model from being fully confident, trading a bit of raw likelihood for
  better calibration.

## 6. Model compression

`compression/quantize_and_benchmark.py` applies PyTorch dynamic quantization
(float32 → int8 on Linear layers) to the trained multilingual model:

| | fp32 | int8 |
|---|---|---|
| Size | 4.55 MB | 1.56 MB (**65.8% smaller**) |
| Latency | 3.49 ms/forward | 2.99 ms/forward |
| Output quality | — | **byte-for-byte identical** greedy output on the spot-check prompt |

## 7. OOP pipeline (`pipeline.py`)

Wraps every stage into a consistent, swappable interface:
- `IndicTokenizer` — thin wrapper isolating callers from the SentencePiece API directly
- `DecodingStrategy` (abstract base) with `Greedy`, `BeamSearch`, `TopK`, `TopP` implementations — swap strategies without touching caller code
- `SafetyFilter` — pluggable accept/reject policy
- `MultilingualGenerationPipeline` — the single entry point composing all of the above

```python
pipeline = MultilingualGenerationPipeline(model, tokenizer, safety_filter)
text = pipeline.generate("<te> విద్యార్థి", strategy=TopP(p=0.9))
```

## How to run everything, in order

```bash
pip install -r requirements.txt

# corpus + tokenizers
python3 data/generate_corpus.py
python3 tokenizer/train_sentencepiece.py     # trains multilingual + per-language tokenizers, prints efficiency comparison
python3 tokenizer/bpe_from_scratch.py        # standalone from-scratch BPE demo

# language model
python3 model/train.py                        # ~1-2 min on CPU
python3 generate.py                            # compare 4 decoding strategies across 3 languages
python3 eval/evaluate.py                       # perplexity, tokenizer efficiency, decoding diversity

# safety-controlled generation
python3 classifier/generate_labeled_data.py
python3 classifier/train.py
python3 classifier/safety_filter_demo.py

# custom training internals
python3 custom_training/custom_optim.py        # sanity check
python3 custom_training/benchmark.py           # full comparison vs. PyTorch built-ins

# compression
python3 compression/quantize_and_benchmark.py

# everything composed via the OOP interface
python3 pipeline.py
```

## What I'd do next with more compute

- Swap the synthetic corpora for real ones (IndicCorp/Samanantar) and re-measure
  whether the tokenizer-efficiency and diversity trends hold at scale.
- Add BLEU/ROUGE against real reference translations for an actual summarization
  or paraphrase task.
- Extend the safety filter beyond sentiment to toxicity/PII detection.
- Try static (not just dynamic) quantization and structured pruning for a bigger
  compression win, and measure on a real accelerator rather than CPU-only timing.
